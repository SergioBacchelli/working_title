# docs/ui — спецификации интерфейса (UI-контракты модулей)

Здесь лежат **UI-спеки** (контракты экранов) для логических модулей.

## Зачем это нужно

UI-спека фиксирует один экран:
- **один логический модуль = один экран**;
- какие есть панели/зоны;
- какие есть поля (ввод/вывод), их ключи и типы;
- какие кнопки/действия есть на экране;
- какие сообщения/коды ошибок может показать экран;
- как данные экрана сохраняются/загружаются (если модуль использует Storage).

Это помогает:
- быстро рисовать UI руками;
- не спорить о названиях полей;
- позже автоматизировать часть генерации интерфейса.

## Где лежат файлы

- `UI_TEMPLATE.md` — шаблон UI-спеки на один экран.
- `UI_FIELD_FORMAT.md` — справочник типов полей и формат параметров (OPTIONS, VALIDATION и т.п.).
- `modules/` — UI-спеки конкретных модулей (по одному файлу на модуль).

Рекомендуемый нейминг файлов модуля:
- `docs/ui/modules/<ModuleId>.ui.md`

## Правила идентификаторов

- `ModuleId` — lowerCamelCase (пример: `primaryCase`).
- `PanelId` — lowerCamelCase (пример: `mainInputs`).
- `FieldKeyEn` — lowerCamelCase (пример: `frontType`).
- `ActionId` — lowerCamelCase (пример: `runCalc`).
- `MessageCode` — UPPER_SNAKE (пример: `ERR_NO_NAME`).

## Минимальный принцип

UI-спека описывает **контракт**, а не конкретную верстку.
То есть мы фиксируем:
- *что есть* (поля/кнопки/сообщения),
- *как оно называется*,
- *какие ограничения/варианты*,
а уже реализация UI решит:
- Tailwind/React/что угодно,
- где именно что стоит,
- какой компонент использовать (checkbox vs toggle).
